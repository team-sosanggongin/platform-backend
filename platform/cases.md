# 엔티티 도메인 관계

1. User의 전화번호는 유일값이다.
    - 소셜 로그인 경로가 다르더라도, 이미 User 정보가 존재한다면, 새로운 User를 테이블에 추가하지 않는다.
    - 소셜 로그인 시, 만약 User 테이블에 유저가 존재하지 않는 경우, 
    - 최초 로그인 시, 휴대전화 인증상태는 완료되지 않았다.
      - 휴대전화 인증에 성공하면, 휴대전화 인증 정보를 저장한다
    - 유저가 생성된 이후, 휴대전화 인증 로직이 실행된다. 이때, 인증정보를 담은 객체는 PhoneVerification이다
2. Business(사업체)
    - 사업체 객체는 반드시 한 명의 대표자를 가지고 있다.
    - 사업체 객체는 그 타입에 따라서, 사업자등록증을 가진 경우와, 그렇지 않은 경우로 나뉜다.
      - REGISTERED
      - INDIVIDUAL
    - 사업체 객체는 다음 세 가지 상태를 가진다
      - ACTIVE 
      - INACTIVE,
    - 사업체는 사업체 메타데이터(BusinessMetadata) 정보를 가진다.
      - 이에는 사업체의 주소, 월급날, 등이 들어올 수 있다.
    - 사업체는 사업자 등록증을 등록하고, 실제 사업장을 인증할 수 있다.
      - 사업체 등록이 필수는 아니지만, 인증이 안된 업체에 대해서는 기능이 제한적으로 열린다.
      - 사업체 등록 시, BusinessRegistration의 상태는 PENDING -> TEMP_VERIFIED -> VERIFIED 순서로 변한다.
        - VERIFIED 상태일 때만, 제한된 기능을 사용할 수 있다.
3. Employment(고용 - 직원과 사업체를 매핑)
   - 유저는 하나 이상의 사업체와 매핑될 수 있다.
   - 매핑된 관계는 크게 3 가지 상태를 가질 수 있다.
     - ACTIVE : 현재 근무중인 상태
     - RESIGNED : 그만둔 상태
     - PENDING : 일시적으로 쉬고 있는 상태
     - VACATION : 휴가중인 상태
   - 사용자는 하나의 사업체에 대하여 1개 이상의 롤을 부여받을 수 있다.(BusinessRole)
     - 이 롤은, 사업자의 사장이 직원을 초대할 떄 부여한다.
     - 롤에 따라 해당 직원이 수행할 수 있는 업무의 범위가 달라진다.
     - BusinessRole은 Business와 매핑되며, BusinessRole은 RolePermission과 매핑되어, 실제 해당 롤에 매핑된 권한을 가지고 있다.
